## 逐文件说明

### 根目录

| 文件 | 作用 | 状态 |
|------|------|------|
| `main.py` | FastAPI 应用入口。启动后提供 `/health` 健康检查和 `/chat` 对话接口。`uvicorn main:app` 启动。 | ✅ `/health` 可用，`/chat` 待 core/llm.py |
| `requirements.txt` | Python 依赖清单。`pip install -r requirements.txt` 一键安装。含 LangChain、FastAPI、Qdrant、RAGAS 等。 | ✅ |
| `docker-compose.yml` | 生产环境编排。`docker compose up -d` 启动 Qdrant + FastAPI + Nginx 三个容器。 | ✅ |
| `docker-compose.dev.yml` | 开发环境编排。只启动 Qdrant 容器，API 在本地 `python main.py` 跑，方便热重载调试。 | ✅ |
| `nginx.conf` | Nginx 反向代理配置。`/api/*` 转发给 FastAPI，`/` 转发给 Streamlit 前端。SSE 流式输出配置已启用。 | ✅ |
| `.env.example` | 环境变量模板，可安全提交 Git。你需要复制为 `.env` 并填入真实 API Key。 | ✅ |
| `.env` | 你的真实环境变量（含 API Key），已被 `.gitignore` 排除，不会提交到 Git。 | ✅ (本地) |
| `.gitignore` | 忽略 `__pycache__`、`.env`、`.venv`、`qdrant_storage` 等不应提交的文件。 | ✅ |
| `CLAUDE.md` | 项目说明书。给 Claude Code 看的技术栈、架构决策、开发命令、每日任务规范。 | ✅ |
| `Dockerfile` | Docker 镜像构建文件。多阶段构建：builder 装依赖 → runner 精简镜像。 | 📋 Day 12 |

### core/ — 核心基础设施

| 文件 | 作用 | 状态 |
|------|------|------|
| `core/config.py` | **全局配置管理**。使用 pydantic-settings 从 `.env` 读取所有配置项（LLM Provider、API Key、Qdrant 地址等）。启动时自动校验：如果 `LLM_API_KEY` 为空 → 报错退出。`get_settings()` 获取全局单例。 | ✅ |
| `core/llm.py` | **LLM 工厂**。`get_llm(config)` 根据 `LLM_PROVIDER`（openai/qwen/deepseek）返回对应的 LangChain ChatModel 实例。三个 Provider 都可走 OpenAI 兼容协议，只需配置不同 `base_url`。 | ⚠️ 待你实现 |
| `core/embedding.py` | **Embedding 工厂**。类似 LLM 工厂，支持 OpenAI Embedding / bge-large-zh / m3e 三种模型的切换。 | 📋 Day 4 |
| `core/observability.py` | **指标收集器**。记录检索延迟 P50/P95/P99、token 消耗、高频 Query Top 20、低质量回答占比。为前端 Dashboard 提供数据。 | 📋 Day 9 |

### rag/ — RAG 核心引擎

| 文件 | 作用 | 状态 |
|------|------|------|
| `rag/models.py` | **数据模型**。定义 `ParsedDocument`（含 sections + tables）、`SectionMeta`（标题层级）、`TableData`（表头+数据行）、`CitedAnswer`（含引用列表）等 Pydantic 模型。所有模块共享的数据结构。 | 📋 Day 2 |
| `rag/loader.py` | **文档加载器入口**。根据文件扩展名分发到对应的解析器（pdf→PDFLoader, docx→DocxLoader, md→MDLoader），统一返回 `ParsedDocument`。 | 📋 Day 2 |
| `rag/loaders/pdf_loader.py` | **PDF 解析器**。PyMuPDF 提取正文 + pdfplumber 提取表格。表格保留为结构化 `TableData`，不摊平成纯文本。 | 📋 Day 2 |
| `rag/loaders/docx_loader.py` | **DOCX 解析器**。python-docx 提取段落和表格。 | 📋 Day 2 |
| `rag/loaders/md_loader.py` | **Markdown 解析器**。按标题层级和 Markdown 表格语法解析。 | 📋 Day 2 |
| `rag/splitter.py` | **切片策略**。含 4 种策略：`RecursiveCharacterTextSplitter`（通用基准）、`MarkdownHeaderTextSplitter`（按标题切）、`SemanticChunker`（语义边界）、**`PolicyClauseSplitter`**（自研核心——识别条款编号保证语义完整）。 | 📋 Day 3 |
| `rag/vector_store.py` | **Qdrant 操作层**。封装创建集合、批量写入向量、Top-K 检索、条件过滤、索引状态查询等操作。 | 📋 Day 4 |
| `rag/indexing_pipeline.py` | **索引流水线**。串联 文档加载→切片→Embedding→写入 Qdrant 的完整流程。支持增量索引（SHA256 hash 对比跳过已索引文档）。 | 📋 Day 4 |
| `rag/bm25.py` | **BM25 关键词检索**。基于 jieba 中文分词的 BM25Okapi 实现，内存索引，支持增量添加文档。 | 📋 Day 5 |
| `rag/hybrid_search.py` | **混合检索器**。并行调用向量检索 + BM25 检索，用 RRF(Reciprocal Rank Fusion) 融合结果。k 值可配置，支持 4 种模式切换。 | 📋 Day 5 |
| `rag/reranker.py` | **重排序模块**。支持 bge-reranker-v2-m3 本地模型和 LLM-based Rerank 两种模式。对 Top-K 结果二次排序。 | 📋 Day 5 |
| `rag/generator.py` | **RAG 生成器 + 引用溯源**。包含 System Prompt 设计、上下文组装、LLM 调用、**条款级引用溯源**（`[来源: 文档名 第X条]`）。自研 `generate_with_citations()` 解析引用标记关联回原文。 | 📋 Day 6 |
| `rag/callbacks.py` | **LangSmith 追踪回调**。标记 RAG 链路的 8 个节点（解析→切片→Embedding→向量检索→BM25检索→RRF融合→Rerank→LLM生成），记录每个节点的耗时。 | 📋 Day 9 |

### agent/ — Agent 路由层

| 文件 | 作用 | 状态 |
|------|------|------|
| `agent/router_graph.py` | **LangGraph 智能路由 Agent**（Day 13 加分项）。包含意图分类（政策查询→RAG 检索 / 操作指引→手册检索 / 其他→LLM 直答）、置信度评估（检索不足时自动改写 Query 重试，最多 2 轮）。 | 📋 Day 13 |

### eval/ — 评估模块

| 文件 | 作用 | 状态 |
|------|------|------|
| `eval/ragas_eval.py` | **RAGAS 评估流水线**。遍历评估集执行完整 RAG 链路，计算 Faithfulness/Answer Relevancy/Context Precision/Context Recall 四项指标。输出评估报告 JSON + 雷达图 + Bad Case 列表。 | 📋 Day 7 |

### api/ — API 接口层

| 文件 | 作用 | 状态 |
|------|------|------|
| `api/routes/chat.py` | 对话接口。`POST /rag/chat`（普通）+ `GET /rag/chat/stream`（SSE 流式）。 | 📋 Day 8 |
| `api/routes/document.py` | 文档管理接口。上传/列表/删除/索引状态查询。 | 📋 Day 8 |
| `api/routes/eval.py` | 评估接口。触发评估 + 查看报告。 | 📋 Day 8 |
| `api/routes/health.py` | 健康检查接口（目前写在 main.py 中，Day 8 迁移到此）。 | 📋 Day 8 |
| `api/middleware.py` | 中间件。全局异常处理（统一错误码 1001~1005）+ 请求日志（耗时/token消耗/检索命中数）。 | 📋 Day 8 |
