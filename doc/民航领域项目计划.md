# 两个垂直项目方向 & 完整 14 天计划（民航领域）

## 方向 A：国航内部员工智能知识助手（RAG 方向 · E 端）

### 业务定位

面向国航**内部一线员工**（客服坐席、值机柜台、登机口、行李查询、特殊服务协调员）的企业级智能知识库。将航司内部的客规运价、行李政策、会员体系、特殊旅客服务流程、航班不正常处置 SOP、证件签证要求、投诉处理规范等海量文档构建为可检索知识库，帮助员工在面对旅客复杂问题时快速获取精准答案。

### 主线聚焦（RAG 闭环）

本项目不以堆叠技术为追求，主线明确为 **RAG 全链路工程闭环**：

```
文档解析质量 → 条款感知切片 → 混合检索 → 引用溯源 → RAGAS 评估 → 工程化部署
```

这 6 个环节构成完整的 RAG 工程叙事，每个环节都有量化评估。LangGraph Agent 路由作为 **Day 13 加分项**，在 RAG 主线稳固后追加，不影响主线完整性。

### 目标用户与典型场景

| 用户角色 | 典型 Query | 知识来源 |
|---------|-----------|---------|
| 客服坐席 | "旅客买的经济舱 Y 舱全价票，起飞前 1 小时退票，手续费多少？" | 客规/运价规则 |
| 值机柜台 | "旅客带了一把小提琴要带上飞机，能放行李架吗？超尺寸怎么办？" | 行李规定/特殊行李 |
| 登机口 | "金卡会员的同行人能一起优先登机吗？最多带几个人？" | 会员权益细则 |
| 行李查询 | "旅客行李箱被摔坏了，赔偿标准是什么？需要什么材料？" | 行李赔偿流程 |
| 特殊服务 | "无陪儿童的年龄范围是多少？国际航班能申请吗？" | 特殊旅客服务手册 |
| 票务柜台 | "CA888 航班取消后，旅客要求改签到东航同航线，能签转吗？" | 不正常航班处置 SOP |

### 简历优势

- RAG 是当前国内大模型应用落地最成熟的赛道，招聘量最大
- **行业垂直 + E 端**：民航内部知识库比通用问答系统有说服力，E 端项目面试演示风险低
- 技术主线清晰，不堆砌：每个技术选型都有量化对比实验支撑
- 覆盖面试高频考点：文档解析（表格感知）、切片策略（条款保护）、混合检索（BM25+向量+RRF）、引用溯源、RAGAS 评估

### A-Day 3：条款感知切片策略

**核心目标**：实现 4 种切片策略，其中 PolicyClauseSplitter 是本项目的核心差异化组件

**代码任务**：
- 实现 `rag/splitter.py`
- 实现 3 种通用切片策略（基于 LangChain 内置但封装为统一接口）：
  1. `RecursiveCharacterTextSplitter`：chunk_size=512, chunk_overlap=50
  2. `MarkdownHeaderTextSplitter`：按 `##` `###` 标题层级切
  3. `SemanticChunker`：使用 Embedding 检测语义边界
- **重点实现 `PolicyClauseSplitter`**（自研核心组件）：
  - 正则识别条款编号模式：`第[一二三四五六七八九十\d]+条`、`\d+\.\d+(\.\d+)?`、`（[一二三四五六七八九十]）`
  - 策略：顶级条款 + 其所有子条款归入同一个 chunk
  - 子条款不独立成 chunk，保证法律条款语义完整
  - 非条款段落按 `##` 标题 fallback 切分
- 实现 `tests/test_splitter.py`：PolicyClauseSplitter ≥ 5 个测试用例
- 实现切片实验脚本：10 份文档 × 4 种策略 → 输出对比 JSON + 4 张直方图 PNG

**可量化验收标准**：

| # | 验收项 | 量化条件 |
|---|--------|---------|
| 1 | 策略实现 | 4 种策略全部可运行，无报错 |
| 2 | PolicyClauseSplitter 条款完整率 | 切分后完整保留「条款编号+所有子条款层级」的条款占比 > 90%（从 142+ 条中抽查 30 条） |
| 3 | 对比报告 | 输出 `splitter_comparison.json`，含每种策略的 chunk 数量、平均长度、长度标准差、条款完整率 |
| 4 | 单元测试 | PolicyClauseSplitter ≥ 5 个用例（正常条款 / 无条款文档 / 混合编号 / 3 层嵌套子条款 / 空文档） |
| 5 | 直方图 | 输出 4 张切片长度分布直方图 PNG |

### A-Day 4：向量存储与 Embedding 选型

**核心目标**：对接 Qdrant，实现全量索引流水线；完成 Embedding 模型对比实验

**代码任务**：
- 实现 `core/embedding.py`：Embedding 工厂，支持 text-embedding-3-small / bge-large-zh-v1.5 / m3e-base 三种切换
- 实现 `rag/vector_store.py`：Qdrant 操作层（创建集合 / 批量 upsert / 检索 Top-K / 索引状态查询）
- 实现 `rag/indexing_pipeline.py`：`文档 → 解析 → PolicyClauseSplitter → Embedding → Qdrant`
- 实现增量索引：文档 SHA256 hash → Qdrant payload 存储 → 重索引时对比 skip
- **Embedding 选型实验**：
  - 15 条民航 query（覆盖 7 个文档类别各 2 条 + 1 条边界 query）
  - 同一批文档分别用 3 种 Embedding 索引
  - 各检索 Top-5，人工标注每条 query 的期望命中 chunk
  - 输出对比表：Precision@5 / MRR / 平均检索耗时

**可量化验收标准**：

| # | 验收项 | 量化条件 |

| 4 | 检索延迟 | 单次 Qdrant 检索 P50 < 50ms, P95 < 200ms（本地环境） |

### A-Day 6：RAG 生成引擎 + 引用溯源

**代码任务**：
- 实现 `rag/generator.py`：

- 实现 SSE 流式输出（FastAPI StreamingResponse）
- 实现多轮对话：`RunnableWithMessageHistory`，历史窗口 = 最近 5 轮
- 实现上下文窗口管理：检索结果 token 超限时按 chunk 相关性降序截断

**可量化验收标准**：

| # | 验收项 | 量化条件 |

## 方向 B：航班不正常处置多 Agent 协作系统（Agent 方向 · B 端）

### 业务定位

航班发生延误/取消/备降时，航司 AOC 需同时协调多个部门。传统串行电话通知平均耗时 40 分钟以上。本系统基于 LangGraph 构建多 Agent 协作框架，实现「事件触发 → 多 Agent 并行协作 → 方案生成 → 人工审批 → 执行」的自动化闭环。

### Agent 边界收敛设计（核心架构决策）

为避免「为 Agent 而 Agent」的质疑，每个节点明确标注其计算性质：

| 节点 | 类型 | LLM 角色 | 判据 |
|------|------|---------|------|
| 事件监控 Agent | **工具调用** | 不需要 LLM | 仅监听航班状态变更事件，触发后续流程。确定性 API 调用即可。 |
| 影响评估 Agent | **LLM 驱动** | 语义理解 | 统计（人数/舱位/转机）是确定性代码；但「从旅客备注中识别特殊需求（婴儿/轮椅/转机紧迫）」需要自然语言理解，规则穷举困难。 |
| 改签方案 Agent | **规则引擎 + LLM 辅助** | 推荐理由生成 | 航班搜索是工具调用，优先级排序是确定性规则（`rules/rebooking.py`）；LLM 仅负责为 Top-3 方案生成推荐理由文本。 |
| 赔偿计算 Agent | **纯规则引擎** | 不需要 LLM | 延误原因 × 时长 × 舱位 → 查表得赔偿。完全确定性计算，LLM 不参与决策。 |
| 资源调度 Agent | **规则引擎 + 工具调用** | 不需要 LLM | 按规则（delay > N 小时 → 需住宿）触发，按规则（头等→5星）分配。确定性匹配。 |
| 通知起草 Agent | **LLM 驱动** | 个性化文案 | 需针对旅客类型（头等/转机/老年）调整语气和内容，模板穷举成本高且僵硬。 |
| 协调总管 Agent | **LLM 驱动** | 综合判断 | 冲突检测是确定性代码，但「升级决策」「优先级权衡」「方案解释」需要 LLM 综合多个因素做判断——**这是 Agent 不可替代价值所在**。 |

**核心论证**：7 个节点中，3 个是纯规则/工具调用（赔偿、资源、事件监控），工作流引擎就能做；3 个需要 LLM（影响评估中的语义理解、通知起草中的个性化、协调总管中的判断）；1 个混合（改签的排序是规则、推荐理由是 LLM）。**协调总管 Agent 是 Agent 架构的不可替代价值**——工作流引擎能执行规则，但无法在检测到冲突后自主判断「这种情况应该怎么权衡、优先级怎么排、要不要升级」。

### 简历优势

- Agent 是 2024-2026 大模型应用最热方向，代表技术前瞻性
- **民航运营场景**：航班不正常处置是航司最高频的复杂运营场景，有真实 SOP 支撑
- 明确区分「确定性规则」和「LLM 决策」——证明你不是把 Agent 当万能药
- LangGraph 并行编排（Fan-out/Fan-in）+ Human-in-the-loop 是高级面试考点

  - Fan-in：4 个子 Agent 全部完成后 → coordinator 汇总
  - ConditionalEdge：审批通过 → notification；审批拒绝 → 调整方案
- 输出 `graph.png` 状态图可视化

### B-Day 6：协调总管 Agent + Human-in-the-loop

**代码任务**：

**协调总管 Agent** (`agents/coordinator_agent.py`，类型：LLM 驱动——Agent 不可替代价值)：
- **确定性部分**（Python 代码）：
  - 冲突检测：改签座位 < 旅客数 → 标记；酒店 < 需住宿 → 标记；转机旅客无衔接 → 标记
  - 升级规则：旅客 > 300 / 延误 > 8h / 国际航线 / 含 VIP → 强制人工审批
- **LLM 部分（不可替代价值）**：
  - 检测到冲突时：生成冲突说明 + 处理建议
  - 多方案各有优劣时：给出推荐意见 + 理由
  - 汇总所有 Agent 输出 → 生成 `coordinator_notes` 供审批参考

**Human-in-the-loop** (`agents/approval.py`)：
- 实现 LangGraph `interrupt()` 断点：协调总管输出后暂停
- 审批 API：`POST /approval/{event_id}/approve` + `POST /approval/{event_id}/reject`
- approve → 继续执行通知 Agent；reject → 回退到方案调整

**可量化验收标准**：

| # | 验收项 | 量化条件 |
|---|--------|---------|
| 1 | 冲突检测 | 构造 3 种冲突（座位不足/房间不足/转机无方案），全部正确标记 |
| 2 | 分级审批 | 小事件(<100 人, <4h, 国内)自动通过；大事件(>300 人或 >8h)强制人工 → 测试各 1 次 |
| 3 | interrupt 暂停 | 流程在 coordinator 后正确暂停，`approval_status=pending` |
| 4 | approve 续流 | `POST /approval/{id}/approve` → 流程继续 → `notifications_sent=true` |
| 5 | reject 回退 | `POST /approval/{id}/reject` → `approval_status=rejected` → 流程回到方案调整节点 |

## 两个项目的技术栈与特性对比

| 维度 | 方向 A：RAG 知识库 | 方向 B：Agent 协同 |
|------|-------------------|-------------------|
| **技术主线** | LangChain + Qdrant + RAGAS | LangGraph + Function Calling |
| **核心差异化** | PolicyClauseSplitter + 条款级引用溯源 | Agent 边界收敛设计 + Fan-out/Fan-in |
| **确定性与 LLM** | 文档处理为确定性，检索与生成为 LLM | 赔偿/资源为纯规则引擎，影响评估/通知/协调为 LLM |
| **评估体系** | RAGAS 4 项指标 + 30 条标测集 | 场景覆盖(3 常规 + 3 边界) + 规则验证(60 组合) + Agent 成功率 |
| **可观测性** | LangSmith RAG 链路 7 节点耗时拆解 | LangSmith Agent 级 Trace |
| **前端风格** | 对话 + 引用弹窗 | 流程看板 + 并行卡片 + 审批面板 |
| **数据资产** | 10 份文档(~23K 字) + 30 条检索测试集 + 30 条 QA 评估集 | 50 航班 + 3×~200 旅客 + 30 酒店 + 30 替代航班 |
